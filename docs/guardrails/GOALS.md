# GOALS.md — このキットの目標の正本（G1〜G14。すべての変更はここを引用する）

> **このキットの存在理由（目的）**: LLM エージェント（Claude Code / Codex / Cline 等）によるアプリ開発を、善意や心得では
> なく機械的な仕組みで——出戻りなく・バグ少なく・速く——回せる状態にすること。
> あらゆる言語・あらゆるアプリで使い回せることを前提とする（G13）。
>
> **このファイルの役割**: その目的を「LLMにとって取り扱いやすい」「バグが減る」
> 「開発時間が短縮できる」の3レンズ＋第4レンズ「意図の保存」＝判定可能な14条に固定した
> **目標の正本**であり、
> 同時にキット（と、それを敷いたリポジトリの開発規約）への変更のレビュー基準。
>
> **運用ルール**: キット・GUARDRAILS.md・バインディングカタログへの変更は、
> コミット/PR本文に**どのGに効くかを1行で書く**。どのGにも効かない変更は入れない
> （「良さそう」は理由にならない——判定できる目標に紐づいて初めて増築できる）。
> 本ルールのうち正本3文書（本ファイル・GUARDRAILS.md・bindings/catalog.md）への変更分は
> commit-msg フックの `governance-without-goal` が機械検査する（§3.4 検査3 — v2.2）。
> 状態列の記号: ✅ = 機構がキットに存在 ／ （列）= 機構は存在し、bindings/catalog.md の
> 採用列の充填で有効化 ／ Gの達成自体は各リポジトリで Step の DoD として実測する。

## レンズ4 — 意図の保存（認知負債の抑制。v2.8 新設＝決定点①を案Aで確定）

コードは「何を」を残すが「なぜ」を残さない——設計根拠が消えた構造は、後続セッションの
LLM（と人間）に再発掘という認知負債を課す。fix⇔テスト（G10＝**回帰の複利**）の対として、
feat⇔plan を「**意図の複利**」で機械強制する。

## この文書自体の運用

- Gの**追加・変更は判定列が書けるものだけ**。判定できない願望はGにしない。
- Gどうしが衝突したら（例: G4の検査追加がG11の予算を食う）、衝突をコミット本文に明記して
  トレードオフの判断を残す——黙って片方を犠牲にしない。
- 移植先リポジトリに本ファイルもそのまま複製する（`missing-required` の対象）。
