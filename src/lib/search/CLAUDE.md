# src/lib/search/ — フォルダ固有の知見

## UIの全文検索は noteSearch.ts(生本文の部分一致)。search.ts の転置索引は非採用経路(2026-07-12)

ユーザー指摘「全文検索が空」への対応: 画面の全文検索(`SearchPanel.tsx`)は
**`noteSearch.ts` の `searchNotesByText` = 現在の全ノート本文をその場で部分一致(大文字小文字
無視)走査**する方式に切り替えた。ノートは全件メモリ上(最大501件)なので即時・常に最新で、
**日本語の部分文字列でも確実に引ける**(下記の転置索引の完全一致制約を受けない。書いた直後で
履歴に未保存でもヒットする)。

`search.ts`(転置索引)＋`indexSnapshot`は**スナップショット保存時に今も索引を作り続けている
が、UIからは今クエリしていない**(`searchSnapshotIds`はテストのみが呼ぶ)。将来「履歴(過去の
スナップショット)を検索」を復活させる時の土台として残置。**普通の全文検索を直す/テストする
なら触るのは `noteSearch.ts` の方**——下のCJK完全一致制約は search.ts 側の話。

## CJKトークナイズの制約(重要——search.ts(転置索引)絡みのテストを書く前に読む)

`tokenize.ts`は分かち書き言語向けの単語境界分割(`\p{L}\p{N}_`の連続)しかしない。
連続した日本語文字列(スペース無し)は**まるごと1トークン**になり、`search.ts`の
転置インデックスは**完全一致**でしか引けない(部分文字列検索ではない)。

```ts
tokenize("検索できるはずのキーワード") // → ["検索できるはずのキーワード"]という1トークン
tokenize("キーワード")                // → ["キーワード"]という別トークン(前者と不一致)
```

このため「長い日本語の一部を検索語にして見つかるはず」というテストは**確実に失敗する**。
検索が絡むテストを書くときは、クエリと完全一致する単独トークンをスペース区切りで
用意すること(例: `"メモ searchable-keyword 本文"`を`forceSnapshot`し、
`searchSnapshotIds("searchable-keyword")`で引く)。

これは製品コードのバグではない——v1の既知の受容済み制約(SPEC.md記載)。直すのは
テストの書き方であって`tokenize.ts`/`search.ts`ではない。
