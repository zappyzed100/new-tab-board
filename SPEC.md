# new-tab-board 仕様書

新規タブを開いたときに専用ダッシュボードを表示する Chrome 拡張。
上部にカスタムブックマークグリッド、下部にオートセーブ + 履歴 + Google Drive 同期付きの複数ノート・メモ帳を持つ、単一ユーザー向けツール。

- ステータス: ドラフト v0.15（v1 スコープ確定）
- 最終更新: 2026-07-08

---

## 1. 概要

### 目的
ブラウザの新規タブを、自分専用の作業起点（ブックマーク + メモ）に置き換える。
データはローカルを主とし、メモは Google Drive に自動バックアップする。

### コンセプト
- **速い**: 新規タブは一瞬で表示される必要があるため、初期描画をブロックする処理を持たない。
- **消えない**: メモは自動で全履歴を保存し、日時から復元できる。加えて Google Drive に自動同期。
- **育つ**: 単なるメモ帳ではなく、複数ノート・履歴 diff・相互リンクで個人ナレッジベースに拡張できる。

### 対象ユーザー
作者本人（自分用ツール）。多人数配布・課金は想定しない。

| 外部 I/O | **Google Drive API**（現行ノートのみ上書き同期、履歴は含めない） / **Google Calendar API（読み取り）**（次の予定カウントダウン用、§4.9） | ⚠ 当初の「local only」から変更。§8 参照 |

### ⚠ I/O 決定の変更点
セットアップウィザードでは「local only（外部サービス無し）」を選択していたが、
メモの Google Drive 自動同期を採用したことで **local + 外部 API** 構成に変更する。
これにより、外部 I/O の seam・ネットワーク系のテストガード・E2E のネットワークモックが必要になる。

### Manifest 要点
- `chrome_url_overrides.newtab` で新規タブページを差し替え。
- 権限: `storage`, `identity`, **`unlimitedStorage`**, `alarms`, `notifications`, `offscreen`（予定前アラーム §4.11）。
  - `unlimitedStorage` は `chrome.storage.local` の約10MB上限を撤廃するだけでなく、ディスク逼迫時の自動削除（eviction）からも除外される。履歴を全保存する本拡張では実質必須。
- OAuth クライアント ID を Google Cloud プロジェクトで発行し、manifest の `oauth2` に登録。
- Drive スコープ（例: `drive.file` — 本拡張が作成したファイルのみに限定するのが最小権限）。

## 4. 機能仕様

### 4.1 ブックマークグリッド

- アイコン + 直下に名称のセルを 10個程度グリッド配置。
- アイコンは favicon 自動取得 + 手動差し替え（絵文字 / 画像）。
- 追加 / 編集（URL・名称・アイコン）/ 削除、ドラッグ&ドロップ並べ替え、名称インライン編集。
- クリックで遷移（同一タブ / 新規タブは設定）。
- 数字キー `1`–`9` でジャンプ。
- 既存 Chrome ブックマークからのインポート（候補）。
- データ配置: `sync`。

---

### 4.2 メモ帳（複数ノート）★

**複数ノート（v1 確定）**
- ノートをタブ or 一覧で切り替え。追加 / リネーム / 削除。
- ピン留め / 並べ替え。

**エディタ / マークダウン（v1 確定）**
- **CodeMirror 6** ベースで素マークダウンを編集（**source of truth は素マークダウン**）。CM6 はスナップショット区切り検出（§4.3）+ エディタ内シンタックスハイライトのために採用。
- **プレビュー（v1）**: `markdown-it`（or `marked`）で HTML 変換 → **`DOMPurify` で sanitize** → React プレビュー用コンポーネントで表示。変更のたびに debounce して再レンダー。表示形式は分割 or トグル（新規タブは縦が狭いためトグル推奨）。
  - チェックボックス = `markdown-it-task-lists`、コードハイライト = highlight.js / Shiki（任意）。
  - 工数感は小（数時間〜1日）。source of truth が素マークダウンのため diff/検索/Drive/NAS は不変。
- **inline live preview（候補 / v1 後）**: `##` がその場で見出しに化ける Obsidian/Joplin 相当。CM6 の decoration（`@atomic-editor/editor` / `codemirror-rich-markdoc` / `codemirror-live-markdown` 等）で後付け可能。カーソル/選択/パース端ケースの磨き込みが必要な難所のため v1 では見送り、素マークダウン基盤の上に後から昇格。
- ProseMirror 系（TipTap / Milkdown）は不採用: 意味構造保持で source of truth が非マークダウン化するため。
- **（最小構成の代替）**: inline を諦めるなら 素 `<textarea>` + markdown-it プレビュー + 手動区切り検出 まで削減可。CM6 を残すと inline への昇格パスが開いたまま。
- `- [ ]` チェックボックス / ToDo。
- `#hoge` 形式のインラインタグ。全文検索・TODO 集約と連携（`#tag` で絞り込み）。
- 各種キーボードショートカット（§6）。

**Google Drive 同期（v1 確定 / 現行ノートのみ）★**
- **現在の各ノートの最新内容だけ**を Google Drive の対応ファイルに上書き同期する（ライブミラー）。
- **履歴（スナップショット）は Drive に入れない。** 履歴はローカル（IndexedDB、§4.3）に保持する。
- これにより Drive 側は常に「現在の内容」のみ。バージョン肥大も、履歴の競合解決も発生しない。
- **ファイル粒度**: ノート1つにつき Drive ファイル1つ（`.md`）。ファイル数の上限は設けない（現実的には ~20 程度でも数百でも可）。Drive 側でも直接読める。
- **同期タイミング**: 入力停止から debounce（例: 2秒）+ blur / タブ非表示時 + 定期。毎キーストロークでは送らない（レート制限回避）。体感は「常に同期済み」。
- **オフライン**: 送信失敗時はローカルにキューし、復帰後に再送。
- **競合**: 現行内容のみのため last-write-wins で単純化可能。
- **同期状態表示**: 同期中 / 同期済 / オフライン をインジケータで明示。

**候補（残り）**
- ログ / 追記モード（各行に自動タイムスタンプ、§9-6）。
- ロック（暗号化）ノート（パスフレーズ、ローカル暗号）。

**データ配置**: 現行内容 = `chrome.storage.local` + Google Drive（ミラー）。履歴 = IndexedDB（§4.3）。

### 4.3 メモ履歴 / バージョン管理 ★核心機能

**保存を2層に分ける（★設計の要）**
- **durability 保存（消えないため）**: 現在の内容を頻繁に上書き保存（`chrome.storage.local` + Drive ミラー）。区切り検出は不要、常時・上書きでよい。履歴は汚れない。
- **history スナップショット（辿る/復元/diff のため）**: 「意味のある編集区切り」でのみ刻む。durability が別にあるので、区切り検出が多少ミスしてもデータは失われない。

**スナップショットのタイミング（編集区切りの自動検出）**
- 「編集区切り」は、エディタが undo をグループ化する境界とほぼ同じ。以下の信号を OR で拾う:
  - アイドル: 入力停止 2〜3 秒（主トリガ）
  - フォーカス喪失 / タブ非表示 / 離脱（`blur` / `visibilitychange` / `pagehide`）
  - ペースト（`paste`。LLM 出力貼り付け等の離散変化）
  - 編集モード転換: 挿入↔削除の切替、カーソルの飛び（`InputEvent.inputType` / `selectionchange`）
  - 変更量 > 閾値（例: 前スナップショットから 200 文字）— 長文連続執筆の安全網
- ガード:
  - 最短フロア（例: 5 秒以内は二重に刻まない）
  - 最長キャップ（アクティブ編集中は最低 60 秒に 1 回）
  - dedup（直前と同一内容はスキップ）
- エディタは CodeMirror 6（§2）。その history が編集を undo グループにまとめるので、**新しい undo グループ境界にフックして snapshot する**のが最短・最もきれい（マークダウン live preview と同じ CM6 を共用）。
- 完璧な検出は不要: 直近は細かく残し、古いものはセッション / 日単位に間引き集約（コンパクション）で掃除する。

**スナップショットの保存方式（★フルスナップショット + オンデマンド diff）**
- **各スナップショットは自己完結のフル本文**（gzip 圧縮）。差分チェーンにはしない。
- 理由: 区切り検出で重複はタイミング側で解決済み、`unlimitedStorage` で容量制約なし、履歴は Drive に流さないので転送量の動機もなし。差分保存の動機が消えている。
- **「差分保存」と「差分表示」は別物**: diff ビューは保存方式に依存せず、2つのフル版から**表示時にその場で計算**する（KB 級テキストの diff は一瞬）。
- フルスナップショットの利点: 復元は1レコード読むだけ、1件破損しても失うのは1版のみ（チェーン破損が無い）、コンパクションは不要レコードの削除だけ、検索は各版の実テキストを直接インデックスできる。
- **将来最適化（先に作らない）**: 実測で容量が問題化した場合のみ、古い（コールドな）履歴に限って後追いの差分圧縮を足す。保存・復元・検索のホットパスは常にフルのまま。

**復元**
- 履歴パネルで日付 / 時刻からスナップショットを選択 → プレビュー → 復元。
- 復元時は現在の本文を先にスナップショットしてから置き換える。

**diff ビュー（v1 確定）★**
- 任意の2スナップショット間の差分を色分け表示。「いつ何を消したか」を可視化。
- 保存はフルのまま、差分は表示時に diff エンジンで算出（diff-match-patch 等）。

**保存バックエンド（★ローカルディスク / IndexedDB）**
- 履歴は `chrome.storage.local`（約10MB上限）ではなく **IndexedDB** に保存する。物理的にはローカルディスク（ブラウザプロファイル領域）。
- `unlimitedStorage` 権限 + `navigator.storage.persist()` で、容量は実質ディスク次第・eviction からも保護。
- 履歴は Drive に上げない（§4.2）。ローカルに無限に溜める前提。

**検索（全文検索）**
- IndexedDB に全文検索機能は無いため、**転置インデックス**（トークン / `#tag` → スナップショットID）を自前で構築し IndexedDB に保持。または FlexSearch / MiniSearch 等の軽量 FTS ライブラリを利用。
- データが小さいうちはカーソル走査 + 部分一致でも実用可。
- 検索対象は本拡張が保存した履歴のみ（HDD 上の任意ファイルはサンドボックス外）。
- ヒット箇所プレビュー + 日時一覧。ノート編集エリア内のトグルとして単独提供（§4.5撤去に伴い統合先は無し）。

**二層アーカイブ（SSD 一次退避 → NAS 本archive / store-and-forward）★**
- 背景: 本archive 先の NAS は電源が落ちることがあるため、常時確実に書ける SSD を一次退避（ステージング）に使い、NAS 復帰時に本archive へ排出する。
- **SSD 一次退避 = IndexedDB**（ローカルプロファイル領域）。スナップショットはまずここへ着地。NAS の生死に無関係にプロンプト無し・確実に書ける。
- **NAS 本archive = File System Access のフォルダ**（NAS マウント先を選択）。バルク長期保管。
- **フラッシュ（新規タブページが開き、NAS ハンドル権限が有効なとき）**:
  1. NAS 到達性を probe（小ファイルの読み書き試行、例外なら「オフライン」判定）
  2. 到達可なら未アーカイブのスナップショット本体を NAS ファイルへコピー
  3. 書き込みを検証（再読込 / サイズ確認）してから
  4. IndexedDB 側の本体を削除（`archived` を立て本体のみドロップ）
- **安全条件**: NAS 書き込みを検証してからしか SSD 側を消さない（データが 0 箇所になる瞬間を作らない）。冪等（スナップショット id キーで再実行しても重複しない）。
- **オフライン時**: probe 失敗なら IndexedDB に溜め続け、復帰時に排出。
- **検索の維持**: 本体を NAS に逃がしても、検索インデックス + メタデータ（日時 / noteId / tags）は IndexedDB に残す。NAS オフラインでも検索は効き、古い版の本文を開くときだけ NAS から取得（オフライン時は degrade 表示）。

**制約（正直な前提）**
- File System Access のフォルダ権限はオリジンのタブが全閉で失効 → 完全無人の常時フラッシュは保証されない。現実解は「新規タブページが開いたタイミングで権限確認 → 有効なうちにフラッシュ」。セッションをまたぐと再許可が要る場合あり。
- File System Access は service worker では不可。フラッシュは新規タブページ文脈で実行。
- 拡張では `showDirectoryPicker` に既知の不安定さ（Chromium バグ）があるため、フォルダ選択は明示的な設定操作として持たせる。

**手動フォルダエクスポート（v1 確定）**
- 全履歴を選択フォルダへ一括書き出し（バックアップ / 移行用）。File System Access の書き込み機構を NAS アーカイブと共用し、シリアライザは JSON エクスポートと重なるため追加コストは小さい。

**データ配置**: SSD 一次退避 = IndexedDB / NAS 本archive = File System Access フォルダ。

  ブックマークグリッドのクリックで引き続き可能（当初の代替先だったコマンドパレット
  §4.5も後日撤去済み）。実装があった記録として節番号は残す。

### 4.5 コマンドパレット / ジャンプ（Cmd+K）（撤去済み・2026-07のユーザーフィードバックで不要と判断）
- 実装していたが「クイックオープン機能消して」との判断で撤去した。ノート切替は
  ノートタブのクリック、ブックマーク遷移はブックマークグリッドのクリック、ファイルを
  開くはデータ管理パネルの「ファイルを開く」で引き続き可能。§4.5.1のURLスキーム
  アプリ起動は、この撤去に伴いアクセス手段が無くなった（実装があった記録として
  節番号は残す）。

#### 4.5.1 アプリ起動（ジャンプの拡張）
- **⚠ Chrome 拡張はネイティブPCアプリを直接起動できない**（サンドボックス）。Alfred がネイティブアプリなのはOSを直接叩けるため。
- **v1 = URL スキーム経由**: `vscode://` / `slack://` / `zoommtg://` / `spotify://` 等、アプリが登録した独自スキームを開くことでOSに起動させる。companion 不要・拡張だけで完結。対応表は手動、スキームを持つアプリに限る。
- **候補（後回し）= 任意アプリ起動（Alfred相当）**: native messaging の companion ホスト（別インストールのローカルヘルパー）が任意アプリ/コマンドを起動。OSごとビルド + セキュリティ面 + インストール手順が増え、「拡張だけ」の手軽さを壊す。将来 NAS/MCP 用に companion を入れるなら相乗り可能。**フォークする Flow Launcher に native messaging host を組み込めば、§4.10-d のファイル橋渡しと同じ host を共用できる**（アプリ起動 + ファイルオープンを1つの host に集約）。

### 4.6 ショートカットマップ（v1 確定）
- `?` キーで全ショートカットの一覧をモーダル表示（Gmail / Linear 系の定番）。
- キーボード駆動が増える（Cmd+K・数字ジャンプ・ノート切替…）ほど「何が割り当たっているか分からない」問題を解く。
- **単一レジストリ駆動**: ショートカットを1箇所で定義し、そこから実際のキーバインドと `?` のチートシートの両方を生成。バインド追加時に一覧が自動同期し、実装と表示がズレない。

### 4.7 JSON エクスポート / インポート（v1 確定）
- 全データ（ブックマーク・設定・ノート・履歴）を JSON で書き出し / 取り込み。
- Drive 自動同期とは別の、手動スナップショット / オフライン退避 / 移行手段として維持。

### 4.8 時計・日付 + テーマ（v1 確定）
- 時計 / 日付表示。テーマ light / dark / auto、カスタム背景。

### 4.9 カレンダー / 次の予定カウントダウン（v1 確定）

**小型カレンダー表示 + クリックで GCal（URL、API不要）**
- 月グリッド、今日ハイライト、前月/翌月切替。外部通信なし。
- 日をクリック → Google カレンダーを **URL 遷移**で開く（API/OAuth/スコープ不要）:
  - イベント作成プリフィル: `https://calendar.google.com/calendar/render?action=TEMPLATE&text=...&dates=YYYYMMDD/YYYYMMDD&details=...&location=...`
  - 終日は `dates=YYYYMMDD/翌日`、`ctz` でタイムゾーン、`recur` で RRULE。時刻あり: `dates=YYYYMMDDTHHMMSSZ/...`
- カレンダーのデータを拡張に取り込まない（プライバシー: GCal に留まる）。一方向（送るだけ）。

**次の予定までのカウントダウン（★最上部・大きく表示 / Calendar API 読み取り）**
- ⚠ **URL 遷移では実現不可**。予定を「読む」ため Calendar API（読み取り）が必須。URL 遷移は一方向で予定を取得できない。
- スコープ = `calendar.readonly`。既存の Drive 用 OAuth に**増分追加**。
- 取得 = `events.list`（`timeMin=now`, `singleEvents=true`〈繰り返しをサーバ側で展開〉, `orderBy=startTime`, `maxResults` 小）。次の予定の開始時刻を得る。
- **ポーリング + ローカルティック**: API は数分ごとに叩いて次予定をキャッシュ、カウントダウンの数字はローカルの時計で毎秒/毎分再計算（毎秒 API は叩かない）。フォーカス / 可視化復帰時にも更新。
- タイムゾーン = API が返すオフセット付き開始時刻で now との差分。
- エッジ状態: 予定なし / 進行中 / 終日（分カウント対象外） / オフライン（最終既知 or ダッシュ）。
- 表示 = new-tab 最上部に大きく「次の予定まで N分」（予定名も併記可）。

**コスト / 位置づけ**: 小型カレンダー + URL 連携 = 〜1日（API不要）。カウントダウン = 〜1.5〜2.5日（認証は増分、狙い撃ちの単一クエリ + ローカルティック。フル・インライン予定表示より軽い）。フル・インライン予定表示（グリッドに点/一覧）は候補送り（§9）。

### 4.10 ローカルテキストファイルを開く / 関連付け

**前提（重要）**: 拡張の `file_handlers` は **ChromeOS 専用**で、**Windows では効かない**。Windows で「.txt をこのアプリで開く/デフォルト」を実現できるのは **PWA の File Handling API**（`file_handlers` + `launchQueue`、要インストール）か **ネイティブ関連付け**のみ。そして new-tab 置き換えは**拡張だけの能力**、.txt デフォルトハンドラは **PWA/ネイティブだけの能力**——**1アーティファクトで両立不可**。

- **(a) アプリ内で開く（v1 確定）**: new-tab から File System Access の `showOpenFilePicker()` で .txt を選び、ノートに読み込み → 編集 → 書き戻し。拡張だけで完結（NAS で使う FSA を流用）。「OS デフォルト」ではなく「ユーザーがアプリ内で開く」形。データ管理パネルに「ファイルを開く」ボタンとして提供（コマンドパレット/omnibarは撤去済み）。
- **(b) OS デフォルト / 「プログラムから開く」（候補）**: 同じノートUIを共有する**相棒 PWA** に `file_handlers` を付け、Windows の Open with に載せ・デフォルト設定可能にする。代償は「拡張（新規タブ）+ PWA（ファイル）の2アーティファクト」。PWA は新規タブを乗っ取れない。
- **(c) ネイティブ companion（候補）**: 関連付けを登録して拡張へ転送。重い（NAS/アプリ起動と同じ橋）。
- **(d) Flow Launcher 経由（候補・推奨の OS デフォルト実現法）★**: アプリ起動用に既にフォークしている Flow Launcher（§4.5.1）を橋に使う。追加コストが小さく、(b)/(c) の上位互換になりうる。
  - **チェーン**: ① Flow Launcher が .txt を読む（native なのでファイルシステム自由アクセス）→ ② `chrome.exe` で新規タブを開く → ③ 拡張が new-tab 起動時に `chrome.runtime.connectNative` で Flow Launcher（内蔵 native messaging host）へ繋ぎ、**待機中のファイル内容を pull** → ノートに読み込み。
  - **URL では不可な理由**: 中身は URL に載らない（長さ制限）、パスだけ渡しても拡張はサンドボックスで任意パスを読めない。→ **実データの搬送は Native Messaging が必須**。
  - **OS デフォルト問題の同時解決**: Flow Launcher は native なので **.txt の OS デフォルトハンドラになれる**。そこから拡張へ転送することで「拡張は OS ハンドラになれない」壁を Flow Launcher が肩代わり。(b) の相棒 PWA を用意せずに済む。
  - **必要事項**: native messaging host マニフェスト（exe パス + 許可する拡張 ID）の登録、拡張側 `nativeMessaging` 権限。既にフォーク配布する Flow Launcher に host を同梱できる。
  - **pull 型にする理由**: MV3 の service worker は寝るため「Flow Launcher が拡張を叩く」は不安定。**Flow Launcher が新規タブを開き、拡張側から接続しに行く**設計にする。
  - **メッセージ上限**: native→拡張は1メッセージ約1MB上限。巨大テキストは**チャンク分割**（通常の .txt は問題なし）。
  - **セキュリティ**: native messaging はマニフェストで**許可した拡張 ID だけ**が接続可。URL スキームの「あらゆる Web ページから叩ける」問題を持たない（相手限定）。

### 4.11 予定前アラーム（v1 確定）

予定の10分前に、**停止ボタンを押すまで鳴り続ける**アラームを鳴らす。

- **データ**: §4.9 で取得済みの次の予定の開始時刻（`nextEventCache`）を使う。追加の Calendar 取得は不要。
- **スケジュール**: `chrome.alarms` で「開始 − 10分」に一発アラームを設定（service worker が寝ていても発火）。次予定が変わったら再スケジュール、取消/移動でキャンセル。
- **鳴動**: MV3 の SW は音を鳴らせないため、`chrome.offscreen`（reason: `AUDIO_PLAYBACK`）でオフスクリーン文書を作り `<audio loop>` でループ再生。
- **停止**: `chrome.notifications` の通知に「停止」ボタン + 新規タブページ上にも大きな停止ボタン。押すとオフスクリーン文書を閉じて停止。任意でスヌーズ。
- **権限追加**: `alarms`, `notifications`, `offscreen`。
- **エッジ**: 終日は対象外、初認識時点で残り10分未満なら即時発火 or 開始時、複数予定は直近のみ、予定キャンセル/移動でアラーム更新。

**⚠ 重要な制約（正直な前提）**: このアラームは **Chrome が起動している間しか鳴らない**。ブラウザを完全に閉じていると発火しない = **「絶対に見逃せない」の保証にはならない**。確実な backstop は**スマホ/OS のカレンダー・リマインダー**（バンド振動と同じ経路）。ブラウザアラームは「PC に向かっている間の駄目押し」と位置づける。

## 5. データモデル

```ts
// --- sync (軽量・端末間同期) ---
type SyncData = {
  bookmarks: Bookmark[];
  settings: Settings;
};

type Bookmark = {
  id: string;
  url: string;
  label: string;
  alias?: string;              // オムニバー/パレットのジャンプ用の短縮語（例:「キニ速」）
  icon: { type: "favicon" | "emoji" | "image"; value?: string };
  order: number;
};

type AppLaunch = {              // §4.5.1 アプリ起動（URL スキーム）
  id: string;
  alias: string;               // 入力語
  scheme: string;              // 例: "vscode://", "slack://"
};

type Settings = {
  openIn: "same" | "new";
  theme: "light" | "dark" | "auto";
  searchEngine: string;        // クイック検索の遷移先テンプレート
  // ...
};

// --- chrome.storage.local (現行内容・キュー) ---
type LocalData = {
  notes: Note[];               // 現在の各ノート（最新内容のみ）
  syncQueue: SyncTask[];       // Drive 送信のオフラインキュー
  nextEventCache?: {           // §4.9 次の予定カウントダウン
    title: string;
    startsAt: number;          // epoch ms（API 取得、ローカルティックで差分計算）
    fetchedAt: number;
  };
};

type Note = {
  id: string;
  title: string;
  content: string;             // 現在の内容
  pinned: boolean;
  order: number;
  driveFileId?: string;        // Drive ミラーファイルの対応付け
  lastSyncedAt?: number;
};

// --- IndexedDB (SSD 一次退避 / 履歴 + 検索) ---
// object store: "snapshots" — 各版は自己完結のフル本文（gzip）
type Snapshot = {
  id: string;
  noteId: string;
  timestamp: number;   // epoch ms
  content?: string;    // gzip 圧縮した本文。NAS 排出後はローカルから削除され undefined になりうる
  archived: boolean;   // true = NAS 本archive へ排出済み（本体は NAS）
  archivePath?: string; // NAS 上のファイルパス
};
// メタデータ + 検索インデックスは archived でも常にローカル保持（オフライン検索のため）
// diff ビューは2つの Snapshot 本文から表示時に算出（保存は差分にしない）

// object store: "searchIndex" — 転置インデックス
type IndexEntry = {
  token: string;               // 単語 or #tag
  refs: string[];              // 含む Snapshot の id 群
};
```

### 配置の使い分け

| データ | 配置 | 理由 |
|--------|------|------|
| ブックマーク / 設定 | `chrome.storage.sync` | 軽量、端末間で揃えたい |
| ノート現行内容 / 同期キュー | `chrome.storage.local` | 小さく即時アクセス |
| ノート現行内容（ミラー） | + Google Drive | 現在の内容のみ上書き、履歴なし |
| 履歴（一次退避）/ 検索インデックス / メタデータ | IndexedDB（SSD） | 常時確実・全文検索・オフライン検索 |
| 履歴本体（本archive） | NAS フォルダ（File System Access） | バルク長期保管。排出後ローカル本体は削除 |

### I/O 抽象モジュール
- `chrome.storage` ラッパ / IndexedDB リポジトリ / Drive クライアント / NAS アーカイバを、それぞれ差し替え可能なインターフェースの背後に置く。
- テスト時はインメモリ storage + フェイク IndexedDB + フェイク Drive + フェイク NAS に差し替え（外部 I/O の seam を集約）。

## 8. 非機能要件 / 制約

- **パフォーマンス**: 新規タブは即座に描画。初期表示は本文・ブックマークのみブロックし、履歴・diff・Drive 同期は遅延。
- **ストレージ**:
  - `sync` 約100KB（軽量データ、8KB/item）。`chrome.storage.local` 約10MB（現行内容・キューのみ）。
  - 履歴（一次退避）は IndexedDB（SSD）。`unlimitedStorage` + `navigator.storage.persist()` で容量はディスク次第 + eviction 除外。各版は自己完結のフルスナップショット（gzip）。差分保存はせず diff は表示時算出（§4.3）。
  - 本体は NAS へ排出し、ローカルは検索インデックス + メタデータを常時保持（§4.3 二層アーカイブ）。
- **容量試算（ヘビー想定: 10h/日、本ツール一本、数百行コピペ含む積極的な開発メモ）**:
  - 純コンテンツ増加 ≈ 200KB/日 ≈ 70MB/年
  - スナップショット churn（生）≈ 15〜25MB/日 → gzip で 3〜5MB/日
  - 全保持（gzip、コンパクションなし）≈ 1〜1.5GB/年（悲観 2〜3GB/年）／コンパクション有 ≈ 100〜200MB/年
  - SSD 一次退避の滞留: NAS が 1 週間落ちても ≈ 30MB、1ヶ月で ≈ 100MB。SSD 負担は無視できる。
  - 結論: NAS・SSD いずれも容量の懸念なし。
- **外部 I/O（Drive）**:
  - OAuth（`chrome.identity`）による認可フロー。トークンの取得・失効・再取得。
  - 最小権限スコープ（`drive.file` 推奨）。
  - **現行ノートのみ上書き**（履歴なし）。オフライン時のキューイングとリトライ、送信の debounce。
  - 履歴を上げないため競合は last-write-wins で単純化。
- **NAS アーカイブ（File System Access）**: フォルダ権限はタブ全閉で失効・`showDirectoryPicker` に既知の不安定さがあるため、無人常時フラッシュは不可。新規タブページが開いたタイミングで権限確認 → 有効なうちにフラッシュ（§4.3）。service worker 不可のため新規タブ文脈で実行。
- **Google Calendar API（読み取り / §4.9）**: 次の予定カウントダウンに使用。スコープ `calendar.readonly` は Drive 用 OAuth への増分。API は数分ごとのポーリングのみ（カウントダウンはローカルティック）。`singleEvents=true` で繰り返しはサーバ側展開。カレンダーの中身を読む権限が増える点はプライバシー上の追加面。クリック→GCal を開く方は URL 遷移のみで API 不要。
- **予定前アラーム（§4.11）**: `chrome.alarms` + `chrome.offscreen`（ループ音）+ `chrome.notifications`（停止ボタン）で実現。**Chrome 起動中のみ発火**し、ブラウザ終了中は鳴らない → 見逃し防止の保証にはならず、確実な保険はスマホ/OS のカレンダー・リマインダー。
- **ローカルファイルを開く / .txt 関連付け（§4.10）**: 拡張の `file_handlers` は ChromeOS 専用で Windows 無効。v1 はアプリ内の `showOpenFilePicker()` のみ。OS デフォルトハンドラ化は new-tab 拡張では不可能。**推奨経路は Flow Launcher（フォーク済み）を native messaging host にして橋渡し**（§4.10-d）: Flow Launcher が .txt を OS デフォルトで受け→新規タブを開き→拡張が `connectNative` で pull。相棒 PWA / ネイティブ companion は代替。
- **アプリ起動**: 拡張はネイティブアプリを直接起動不可。v1 は URL スキーム遷移のみ。任意アプリ起動は native messaging companion が必要で候補扱い（§4.5.1）。
- **プライバシー**: 一次データはローカル。Drive ミラーは本人アカウントのみ。ロックノートはローカル暗号。
- **信頼性**: 履歴は SSD(IndexedDB) → NAS の二層 + JSON エクスポートで担保。NAS 書き込み検証後のみ SSD 側を削除（0 箇所化を防ぐ）。
